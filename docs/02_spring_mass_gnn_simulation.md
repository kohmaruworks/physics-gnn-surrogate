# 第2章：物理系の圏論的モデリングとAI学習の実践

本章では、Julia の Catlab.jl によって構築した 1 次元バネ–質量系のグラフ表現と、PyTorch Geometric（PyG）によるグラフ畳み込みネットワーク（GCN）の学習パイプラインを一連の実装として説明する。読者は、AI および数理シミュレーションの橋渡しとして、**圏論的データ構造から JSON 経由で深層学習まで**をどのように接続するかを把握できることを目標とする。

対象となるコードは、ワークスペース内の `spring_mass_chain_export.jl`（グラフ生成と JSON エクスポート）および `train_spring_mass_gcn.py`（データ読み込み・GPU 転送・学習）である。JSON の入出力形式は `export_catlab_graph_json.jl` と `import_catlab_json_to_pyg.py` に依存する。

---

## 1. バネ-質量系の圏論的モデリングと双方向グラフの採用理由

### 1.1 Catlab.jl によるグラフとしての記述

1 次元の直列バネ–質量系では、質点を **頂点**、質点間を結ぶバネによる **近接関係** を **辺** として離散化する。本実装では Catlab の `Catlab.Graphs.Graph` を用いる。これはスキーマ上 **有向マルチグラフ**（辺に識別子があり、同一の頂点対に複数辺を許す）として定義されるデータ構造である。

`spring_mass_chain_export.jl` 内の `spring_mass_chain_graph(n)` は、質点数 `n` に対し頂点を `n` 個追加し、隣接ペア `(i, i+1)`（`i = 1, …, n-1`）について **2 本の有向辺** を張る。具体的には、`add_edges!` に渡すソース列とターゲット列が

- `(i → i+1)` と `(i+1 → i)`

のペアを繰り返す形になっており、`n = 5` のとき辺数は `4 × 2 = 8` となる。

### 1.2 なぜ「双方向の有向辺」とするのか

ニュートン力学において、2 質点を結ぶ理想バネが及ぼす力は **作用・反作用の対称性** を満たす。すなわち、質点 `i` が質点 `i+1` に与える影響と、その逆方向の影響は、表現を分けて記述するのが自然である。無向グラフとして 1 本の辺だけを置く設計も可能であるが、PyG を含む多くの GNN 実装は **有向の `edge_index`**（ソース頂点・ターゲット頂点の列）を前提とする。そのため、物理的に無向に近い結合を表す際には、**無向辺を 2 本の逆向きの有向辺に分解する**のが一般的である。

グラフニューラルネットワークの **メッセージパッシング** は、概ね「各辺 `(s → t)` について、送信ノード `s` の表現からメッセージを生成し、受信ノード `t` で集約する」という非対称な手続きとして記述される。双方向辺を明示しておくと、**隣接ノードから自分へ向かう方向の辺**と**自分から隣へ向かう方向の辺**がデータ上区別され、層ごとの集約が「誰から情報が来たか」のグラフ構造と整合的になる。単方向のチェーン（例：`1→2→…→5`）のみとした場合、ある頂点から見た「上流」「下流」が非対称になり、バネ結合の **局所かつ対称な結合** という物理像とはずれやすい。

以上より、本実装は **圏論的グラフ（有向マルチグラフの枠組み）に、物理的対称性と GNN のメッセージ方向性の両方を載せる** という設計判断を採用している。

### 1.3 ノード特徴とエクスポート

各質点 `i` について、初期位置と初速度を **2 次元ベクトル** として行列 `x` の第 `i` 行に格納する（`randn` による無次元のランダム値）。`save_catlab_graph_json` により、グラフ構造と `x` が中間 JSON（本ワークスペースでは `spring_mass_chain_5.json`）に書き出される。頂点 ID の 0-based 化など、Julia 側と Python 側の索引の対応は第1章系のブリッジ設計（`export_catlab_graph_json.jl` / `import_catlab_json_to_pyg.py`）に従う。

## 2. GCN（グラフ畳み込みネットワーク）の構造と設計意図

### 2.1 ネットワーク全体像

`train_spring_mass_gcn.py` で定義される `TwoLayerGCN` は、PyG の `GCNConv` を 2 段重ねた **ノード単位の回帰** 用モジュールである。

| 層 | 演算 | 入出力チャネル |
|----|------|----------------|
| 1 層目 | `GCNConv` | 2 → 16 |
| 活性化 | `ReLU` | 16 → 16 |
| 2 層目 | `GCNConv` | 16 → 2 |

入力チャネル数 2 は、JSON に保存した **位置・速度の 2 特徴** に一致させる。出力チャネル数 2 は、ダミー教師信号も同じ形状（ノードあたり 2 次元）とするためである。隠れ次元 16 は、**局所結合だけでは線形結合に近い表現力に留まりうる GCN に、中間の非線形表現空間を確保する**ための余裕幅として選ばれた典型的な規模である（本稿の実装ではハイパーパラメータとして固定値を用いる）。

### 2.2 `GCNConv` における特徴の変換と集約

各 `GCNConv` 層は、大まかに次のロジックで動作する。

1. **近傍からの集約**  
   各頂点 `v` について、グラフ上で `v` に入射する有向辺 `(u → v)`（実装では `edge_index` の列が `(u, v)` を指す）に沿って、隣接頂点 `u` の現在の特徴ベクトルを参照する。

2. **正規化付きの線形結合**  
   Kipf & Welling 型の GCN では、次数に基づく正規化係数を伴い、**自分自身の特徴**と**隣接頂点の特徴の加重和**が 1 つの線形層に入力される形にまとめられる（PyG の `GCNConv` は自己ループの追加や正規化を内部で扱う）。

3. **学習可能な線形変換**  
   集約されたベクトルに、チャネル次元にわたる学習可能な重みが掛けられ、次の層への入力となる。

第 1 `GCNConv` のあと `ReLU` を挟むことで、**局所的に集約された 16 次元表現に非線形性**を与える。第 2 `GCNConv` は、その表現を再度隣接構造に沿って集約しつつ、**出力 2 次元へ射影**する。

物理的イメージで言えば、各質点は **自分の位置・速度** に加え、**バネで直結された隣接質点の（中間層での）状態** を繰り込んだ上で、次のベクトル値を出力する。本ダミー学習ではその出力を教師と MSE で合わせるが、応用では力学的な次ステップや観測量の予測に差し替えられる。

## 3. GPU（CUDA）駆動による学習ループの実装解説

### 3.1 JSON から `Data` へ、およびデバイス転送

Julia 側で生成された `spring_mass_chain_5.json` は、`import_catlab_json_to_pyg.py` の `catlab_json_to_data` によって `torch_geometric.data.Data` に復元される。

- `edges` のリスト `[[s,t], …]` は、0-based の頂点番号として解釈され、`torch.tensor` 後に転置され **`edge_index`**（形状 `[2, num_edges]`、`dtype` は既定で `int64`）となる。
- `num_nodes` は `Data` の `num_nodes` に設定され、孤立頂点のない本チェーンでもグラフサイズが明示される。
- `x` が存在する場合は **`float32` のテンソル** として `data.x` に格納される（形状 `[num_nodes, 2]`）。

`train_spring_mass_gcn.py` の `main` では、まず `catlab_json_to_data(json_path)` で CPU 上に `Data` を構築し、続けて

```python
device = torch.device("cuda" if torch.cuda.is_available() else "cpu")
data = data.to(device)
```

とする。`Data.to(device)` は **`x` と `edge_index` をまとめて** 指定デバイスへ移す。CUDA が利用可能な環境では GPU 上で畳み込みと損失計算が実行され、利用不能な場合は CPU にフォールバックする。運用上は、実行ログの `device: cuda` または `device: cpu` で実際の実行場所を確認できる。

同様に、モデルは `TwoLayerGCN(...).to(device)` で同一デバイスに配置する。入力 `data.x` とパラメータが同じデバイスにあることが、PyTorch の実行条件となる。

### 3.2 ダミー教師、損失、最適化

本実装は物理シミュレーションそのものではなく、**パイプライン検証**を目的とする。教師信号は

```python
torch.manual_seed(7)
target = data.x + 0.05 * torch.randn_like(data.x)
```

として **入力特徴に固定のガウスノイズを加えたテンソル** とする。学習中に `target` を更新しないため、ネットワークは時間とともにこの決まった写像に近づくよう勾配降下される。

損失関数は `nn.MSELoss()`（平均二乗誤差）である。各ノードの 2 次元出力と `target` の差の二乗平均がスカラー `loss` となり、`loss.backward()` で全パラメータに勾配が伝播する。

最適化には `torch.optim.Adam(model.parameters(), lr=0.02)` を用いる。Adam はモーメントと適応的ステップにより、素朴な勾配降下より安定しやすい。学習率 `0.02` は本スクリプト固有の選択であり、実データに合わせて調整するのが望ましい。

### 3.3 100 エポックの学習ループ

各エポックで次を繰り返す。

1. `model.train()`  
   ドロップアウト等を使う場合のモード切替（本モデルでは実質的に慣例的呼び出し）。

2. `optimizer.zero_grad(set_to_none=True)`  
   前ステップの勾配を破棄する。`set_to_none=True` はメモリ効率のための PyTorch の推奨パターンである。

3. **順伝播**  
   `out = model(data.x, data.edge_index)`  
   2 層 GCN がグラフ構造を用いて `out`（`[5, 2]`）を計算する。

4. **損失**  
   `loss = criterion(out, target)`

5. **逆伝播**  
   `loss.backward()`

6. **パラメータ更新**  
   `optimizer.step()`

7. **監視**  
   `print(f"{epoch}\t{loss.item():.6f}")` により、エポック番号と現在の MSE を出力する。

100 回繰り返すことで、損失が初期値から低下し、ターゲットに対するフィッティングが進む。実測では環境依存があるが、損失が単調減少しないエポックが混じることもあり、これは学習率やターゲットのスケールに起因しうる。長期的には MSE が小さく落ち着く傾向が確認できる。

### 3.4 実行順序の整理

1. `julia --project=. spring_mass_chain_export.jl` で `spring_mass_chain_5.json` を生成する。  
2. `.venv/bin/python train_spring_mass_gcn.py`（または同等の Python 環境）で学習を実行する。

以上が、**圏論的グラフ → JSON → PyG → CUDA 上の GCN 学習** という一連の実装の技術的要点である。
